How I’m Measuring

ICMP Clocksync

When a client sends a timestamp request:

  1. Client records local send time \(t_0\)
  2. Server receives packet and writes its receive time \(t_1\)
  3. Server sends reply and writes transmit time \(t_2\)
  4. Client receives reply at time \(t_3\)

It’s easy with clockdiff!

╰─○ sudo clockdiff case.edu  
..................................................
host=case.edu rtt=53(11)ms/31ms delta=14ms/14ms Tue Feb 24 12:01:05 2026

I made a zmap module to scan the web for ICMP Timestamp respondees by forking their ICMP ping example module. Then I got a zmap allowed VPS, and I am working on doing an internet scan!

Derivation

And you do math to figure out the time

Then

\(\text{RTT} = t_3 - t_0\)

\(t_1 = t_0 + d_1 + \text{offset}\)

\(p = t_2 - t_1\)

\(t_2 = t_1 + p\)

\(t_3 = t_2 + d_2 - \text{offset}\)

\(\text{offset} = \frac{(t_1 - t_0) + (t_2 - t_3)}{2}\)

  1. Send many requests right at the edge of the second
  2. So you send at \(0.995, 0.996, 0.997, 0.998, 0.999, 1.000, 1.001, \ldots\)
  3. See which was the first for the Date: to change. Let’s say we received it at \(0.998\) in our time
  4. Assume symmetric network latency (not true, but NTP also assumes this, and hopefully it’s good enough). Let’s say \(\text{rtt}=0.1\)
  5. So, \(0.998 - \frac{\text{RTT}}{2} = 0.998 - 0.05 = 0.948\) is when the server sent the response
  6. If we assume the server’s time is exactly \(1.000\) when it sent it, the offset is \(1.000 - 0.948 = 0.052 = 52 ms\)
